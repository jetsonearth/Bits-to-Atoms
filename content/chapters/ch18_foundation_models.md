## Foundation Models 进入机器人 - VLA 和具身智能

你大概经历过这样的时刻：第一次用 ChatGPT 写代码，第一次用 Midjourney 出图，第一次看到 GPT-4 理解一张照片里的笑话。每次都觉得“世界变了”。

现在想象一下，你对机器人说“帮我把桌上的脏碗收进洗碗机”，它看了一眼桌子，自己判断哪些是脏碗、洗碗机在哪、门怎么开，然后一个个放进去。不是因为你预编程了每一步，而是它“理解”了这句话，“看到”了场景，“决定”了动作。

这就是 Foundation Models 进入机器人领域正在尝试做的事。如果说前面几章讲的 SLAM、Nav2、MoveIt 是机器人的“小脑” - 处理具体的走路和抓取，那这一章讲的是给机器人装一个“大脑” - 一个能理解语言、看懂世界、做出决策的通用智能层。听起来很美好，但请先别激动，我们得诚实地聊聊它到底走到哪了。

---

### 从 LLM 到 VLA：一条逐步“长出身体”的进化路线

要理解 VLA，先看三个缩写的递进关系：

**LLM（Large Language Model）** - 只处理文字。你给它一段文本，它输出一段文本。GPT-4、Claude、Qwen 都是这个。它能做推理、做规划、写代码，但它对物理世界一无所知 - 它不知道一个杯子拿起来有多重，也不知道桌子边缘在哪。

**VLM（Vision-Language Model）** - 处理图像 + 文字。你给它一张照片加一个问题“桌上有什么？”，它能回答“一个红色杯子和一本书”。GPT-4o、Qwen-VL 都是这个。它能“看”了，但它不能“动” - 它的输出还是文字。

**VLA（Vision-Language-Action）** - 处理图像 + 文字 + 输出动作。你给它机器人的摄像头画面和一句指令“把红色杯子放到左边”，它直接输出机器人关节应该怎么动。这是真正的从感知到行动的闭环。

这三步的核心思路是一样的：用 Transformer 架构统一不同模态的信息。LLM 把文字变成 token 序列来处理，VLM 加上了图像 token，VLA 则把机器人的动作也编码成 token - 关节角度、末端执行器位置这些连续值被离散化成一串“动作 token”，和文字 token、图像 token 一起丢进同一个模型训练。

这个思路最早被 Google DeepMind 的 **RT-2** 在 2023 年验证。RT-2 直接在 PaLI-X 这样的大规模 VLM 上做 fine-tune，把机器人的动作表示成文本 token（比如 “1 128 91 241 5 101 127”），让模型像“说话”一样“输出动作”。结果很惊人：模型不仅能执行训练数据里见过的任务，还展现了一定的泛化能力 - 比如面对一个从没见过的物体，它能根据语言指令“把最小的物体移到旁边”来正确操作。

### 这个领域的玩家都有谁呢？

这个领域的玩家和模型在过去两年爆发式增长，列几个最有代表性的：

**Google DeepMind RT 系列**。RT-2 是先驱，证明了 VLA 路线可行。后续的 **RT-H**（Robot Transformer with Hierarchies）引入了动作层级 - 不是直接从语言指令跳到关节角度，而是先预测中间层的“语言动作”（比如“向前移动手臂”、“向右旋转”），再从这些语言动作预测低层动作。这种层级结构在多任务学习场景下效果明显更好，比 RT-2 提升了 15% 的成功率。

**NVIDIA GR00T N1 系列**。NVIDIA 在 2025 年 3 月发布了 GR00T N1，定位为“第一个开放的人形机器人基础模型”。它的架构是双系统设计：一个快速的 action model 负责实时控制和动作生成（跑在机器人本体上），一个慢速的 VLM 负责场景理解和任务规划。到 N1.5 版本，在 DreamGen 仿真基准上的成功率从 13.1% 提升到 38.3%。最新的 **N1.6**（2026 年 CES 发布）进一步优化了架构和数据管线。NVIDIA 的策略很明确：做机器人 AI 的“Android” - 模型开源，但你得用它的 Isaac 仿真平台来训练和部署。

**银河通用（Galbot）GraspVLA / GroceryVLA**。这是一个很有意思的案例。GraspVLA 是第一个专门针对抓取任务的 VLA 基础模型，训练数据来自 SynGrasp-1B - 一个十亿帧级别的仿真合成数据集。它能做到零样本抓取：给一个从没见过的物体，不需要额外训练就能成功抓起来。在此基础上，**GroceryVLA** 针对零售场景做了特化 - 在 2025 年 WAIC 上展示了能处理数百种不同商品的自主上货能力。这说明 VLA 的一个现实落地路径可能不是“通用万能”，而是在垂直场景里先打透。

**宇树（Unitree）UnifoLM 系列**。宇树在 2025-2026 年发布了两个模型：**UnifoLM-WMA-0** 是一个 World Model - Action 架构，核心是一个世界模型，既能当仿真器生成合成数据，又能通过预测未来交互来优化决策。**UnifoLM-VLA-0**（2026 年 1 月发布）在 LIBERO 仿真基准上拿到了 98.7 分 - VLA 类别最高分，超过了 OpenVLA、InternVLA 和 Physical Intelligence 的 pi-0 系列。两个模型都已开源，这对于一家硬件厂商来说是个大胆的策略。

### LLM 在机器人上的实际用法：不是直接控制关节

看完上面这些 VLA 模型，你可能觉得未来的方向就是一个端到端的大模型搞定一切。但在 2026 年的工程现实里，LLM/VLM 在机器人上最成熟的用法不是 VLA 那种直接输出动作，而是做**高层任务规划和自然语言交互**。

具体来说，三种模式按成熟度排序：

**模式一：任务规划器**。用户说“帮我收拾桌子”，LLM 把它拆解成：“1. 检测桌上所有物体 → 2. 分类（垃圾 vs 餐具 vs 食物）→ 3. 垃圾扔进垃圾桶 → 4. 餐具放进水槽 → 5. 食物放回冰箱”。每一步调用已有的技能模块（感知、导航、抓取），LLM 本身不碰底层控制。这是目前最成熟、最常用的模式。

**模式二：自然语言交互层**。机器人在执行任务时遇到歧义 - “把那个东西放到那边”，“那个东西”是什么？“那边”在哪？LLM 结合视觉输入做消歧，或者直接问用户“你是说红色的杯子吗？”。这给了机器人一种“常识推理”的能力，不再需要用户给出精确到坐标的指令。

**模式三：VLA 端到端控制**。就是上一节讲的，直接从视觉 + 语言输出动作。效果越来越好，但在可靠性和安全性上离生产部署还有距离（下面会细说）。

大部分团队目前的实际做法是模式一或模式二：LLM 做“大脑”负责理解和规划，然后通过 API 调用传统的感知、导航、操作模块来执行。这样既利用了 LLM 的语言理解能力，又不把安全攸关的底层控制交给一个不确定性很高的模型。

### MCP（Model Context Protocol）与机器人：让 LLM “长出手脚”的标准协议

如果 LLM 要调用机器人的技能 API，那就需要一个标准的“接口协议”。这就是 MCP 的角色。

你可能已经在 AI 工具链里接触过 MCP - 它是 Anthropic 在 2024 年推出的开放协议，定义了 LLM 怎么发现和调用外部工具。在软件开发场景里，MCP 让 Claude 能读文件、查数据库、调 API。在机器人场景里，这些“工具”变成了 move_to_position、detect_objects、pick_up、navigate_to 这样的机器人技能。

最近一年这个趋势加速明显。几个典型案例：

**宇树 Go2 MCP Server**。这是一个开源项目，在宇树 Go2 机器狗上搭建了 MCP 服务器。用户对 LLM 说“走到沙发旁边然后坐下”，LLM 通过 MCP 调用 Go2 的 ROS2 服务来执行 - 自然语言被翻译成具体的运动指令。代码开源在 GitHub 上，用 Apache-2.0 协议。

**ROS 2 Bridge MCP Server**。一个更通用的方案，把 ROS 2 的 service 和 topic 直接暴露为 MCP 工具和资源。这意味着任何 MCP 兼容的 AI 工具（Claude、GPT 等）都能直接和 ROS 2 机器人对话，做到发布运动指令、读取传感器数据、触发导航任务。

**DimOS 的 MCP 集成**。DimOS 内置了 MCP server，把机器人的核心能力 - 导航、检测、移动、视觉伺服等 - 封装成标准的 MCP 工具。这样任何 MCP 兼容的 AI agent 都能直接调用这些技能，不需要自己去对接底层的 ROS 2 话题和服务。它本质上是在 LLM 和机器人硬件之间加了一层标准化的抽象。

MCP 在机器人上的意义远不止“方便调用”。更深层的价值是**可组合性** - 你可以把不同来源的技能（Nav2 的导航、MoveIt 的操作、自定义的感知流水线）统一暴露给 LLM，让 LLM 根据任务需要自由组合。这比硬编码一棵行为树灵活得多，虽然目前在可靠性上还不能完全替代行为树。

### 现实 check：demo 很酷，离可靠部署有多远？

到这里你可能已经很兴奋了。但作为写过生产代码的人，你一定会问：这些东西到底能不能用？

诚实的回答是：**demo 级别已经令人印象深刻，但离 production-ready 还有实实在在的距离。**

几个核心问题：

**可靠性不够**。NVIDIA GR00T N1.5 在仿真基准上的成功率是 38.3%。就算 N1.6 大幅提升，想想你的产品经理听到“我们的机器人做这个任务有 60% 的概率能成功”会是什么表情。工业场景需要的是 99% 以上的可靠性，一次失败可能意味着砸坏一件商品、伤到一个人、或者整条产线停摆。VLA 在受控的实验室环境里表现不错，但真实世界的长尾 case - 光照突变、从没见过的物体形状、传感器噪声 - 还是它们的致命软肋。

**推理延迟**。VLA 模型通常需要几百毫秒到几秒的推理时间。对于“把杯子放到桌上”这种不紧急的任务还好，但对于需要实时反应的场景（机器人在走路时有人突然挡路、抓取过程中物体滑落），这个延迟是不可接受的。这也是为什么 GR00T N1 采用双系统设计 - 快系统处理实时控制，慢系统处理规划。

**安全边界模糊**。传统机器人系统有明确的安全约束：关节力矩限制、速度限制、碰撞检测急停。这些都是确定性的规则。当你把控制权交给一个概率模型，怎么保证它不会输出一个把手臂甩到人脸上的动作？目前的做法通常是在 VLA 外面包一层安全过滤器，但怎么定义“安全”本身就是个开放问题。

**数据饥渴**。训练一个好的 VLA 模型需要海量的机器人操作数据。但机器人数据的采集成本远高于互联网文本和图像 - 你需要真实的硬件、真实的场景、真实的操作。仿真合成数据（比如 GraspVLA 的 SynGrasp-1B 数据集）是目前的主要解法，但 sim-to-real gap 仍然存在。

**踩坑场景**：一个团队用某 VLA 模型做桌面物品整理的 demo，在实验室里成功率 85%，换了一张不同颜色的桌子掉到 40%，换了一个不同角度的摄像头掉到 20%。最后发现模型严重过拟合了训练时的视觉背景和相机视角。他们不得不回到传统感知 + 规划 + 控制的流水线，只在最上层用 LLM 做任务规划。这是目前大部分实际落地项目的真实写照。

所以对 AI 开发者来说，最务实的策略是：**用 LLM/VLM 做高层理解和规划（模式一和模式二），用传统模块做底层执行，同时持续关注 VLA 的进展。** 这个领域的发展速度是非线性的 - 去年觉得“十年后才能实现”的东西，今年可能就出了一篇论文把成功率翻了一倍。但在把它部署到真实产品之前，永远要问自己：失败的最坏后果是什么？你能承受多少不确定性？

Foundation models 进入机器人是不可逆的趋势，但它重新定义“机器人开发”这件事的方式，**可能不是替代 SLAM、Nav2、MoveIt 这些传统模块，而是在它们之上加一层理解和推理的能力。** 接下来几年最有价值的技能，可能恰恰是同时理解这两个世界的人 - 既懂 foundation model 的能力边界，又懂机器人系统工程的安全约束。
