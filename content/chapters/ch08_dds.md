## DDS 深入 - 机器人的通信高速公路

上一章我们把 ROS 2 比作一家公司，Node 是员工，Topic 是群聊频道。但我们跳过了一个关键问题：这些群聊消息到底是怎么送达的？用的是微信还是邮件？如果网断了怎么办？消息能保证不丢吗？

想象一下你在管理一个大型仓库。仓库里有几十台设备同时运行 - 叉车、机械臂、AGV、安全摄像头。它们之间需要不停地交换信息："我在这个位置"、"前方有障碍物"、"去 3 号货架取货"。现在问题来了 - 你用什么通信方式？微信群？太慢了。对讲机？没法传图像数据。电话？一对一的，没法广播。

你需要的是一套能同时处理几十路通信、支持不同优先级、允许新设备随时加入、而且不需要一个"中央交换机"的通信系统。这就是 **DDS（Data Distribution Service）** 干的事。

---

### DDS 是什么，ROS 2 为什么选了它

DDS 是 OMG（Object Management Group）定义的一套分布式数据通信标准，最早用在军事和航空航天领域 - 战斗机编队、军舰作战系统这种对通信可靠性要求极高的场景。它的核心设计哲学是 **去中心化的发布-订阅通信**。

ROS 1 的通信依赖一个叫 `rosmaster` 的中心节点。所有节点启动时都要去 master 那里"注册"，master 负责撮合发布者和订阅者。问题很明显：master 挂了，整个系统就瘫了。而且 master 是单点瓶颈 - 节点多了之后，所有的发现请求都要经过它。

ROS 2 选择 DDS 作为底层通信中间件，最核心的原因就是 **去掉了这个 master**。DDS 用一套叫 RTPS（Real-Time Publish-Subscribe）的协议，让节点之间自动发现彼此 - 不需要中央协调者。你启动一个新节点，它会在网络上广播"我来了，我发布 /scan 话题"，其他对 /scan 感兴趣的节点自动连上。有点像 AirDrop - 你打开它，附近的设备自动出现在列表里。

除了去中心化，DDS 带来的另外两个关键能力是：

**QoS 策略** - 你可以精细控制每一路通信的"服务质量"。传感器数据丢一帧没关系，导航指令一帧都不能丢 - DDS 让你针对每个话题单独配置这些策略。下一节详细讲。

**安全机制** - DDS-Security 规范支持身份认证、访问控制和数据加密。在仓库里跑的 demo 可能不在乎安全性，但如果你的机器人要过 CE 认证或者进医院，通信安全是硬性要求。

对开发者来说，好消息是 DDS 这一层大部分时候是透明的 - 你还是写 `publisher.publish(msg)` 和 `create_subscription(...)`，不需要直接操作 DDS API。但当通信出问题的时候（消息延迟、丢包、收不到数据），你必须理解 DDS 在干什么，才知道往哪里排查。

---

### QoS - 消息到底要不要保证送达

**QoS（Quality of Service）** 是 DDS 最实用也是最容易搞混的部分。简单说，QoS 就是你跟通信系统之间的一份"服务协议" - 你告诉它这条消息有多重要、要不要保证送达、过期了怎么办。

ROS 2 暴露了好几个 QoS 维度，最常用的有四个：

**Reliability（可靠性）** - 这是最关键的一个。

- **Best Effort**：尽力送，丢了就丢了。适合高频传感器数据 - LiDAR 一秒出 10-20 帧，丢一帧完全不影响。重传反而添乱，因为等重传的那帧到了，新数据早就来了。
- **Reliable**：必须送到，丢了就重传。适合控制指令 - "停下来"这条命令如果丢了，机器人就撞墙了。导航目标、急停信号都用这个。

**Durability（持久性）** - 新订阅者能不能收到它"加入之前"发布的消息？

- **Volatile**：不能。你加入频道之后才能看到新消息，之前的历史消息看不到。
- **Transient Local**：能。发布者会把最近的消息缓存住，新订阅者一连上就先收到"最后一条"。这对地图数据很重要 - 如果你的定位节点比建图节点晚启动，它需要拿到已经建好的那张地图。

**History（历史）** - 缓存多少条消息。

- **Keep Last(N)**：只保留最近 N 条。传感器数据通常设 Keep Last(1) - 你只关心最新的那一帧。
- **Keep All**：全部保留。很少用，因为内存会爆。

**Deadline** - 发布者承诺的最大发布间隔。如果超过这个时间没有新消息，订阅者可以触发一个回调来处理"数据断了"的情况。用在安全相关的场景 - 如果机器人超过 200ms 没收到传感器数据，应该立刻减速或停下。

**最容易踩的坑：QoS 不匹配。** ROS 2 有一条铁律 - 发布者和订阅者的 QoS 策略必须兼容，否则连接根本建立不起来，消息一条都收不到。最经典的情况：发布者设了 Best Effort，订阅者设了 Reliable。订阅者说"我要求保证送达"，发布者说"我只能尽力而为" - 谈不拢，DDS 直接不连。

更恼人的是，**这种不匹配不会报错，只是静默失败** - 你的订阅者就是收不到消息，终端里不会有任何 error log。你可能会花一个小时检查 topic 名字有没有拼错、消息类型对不对，最后发现是 QoS 没对上。

排查方法：`ros2 topic info /your_topic --verbose` 会显示每个发布者和订阅者的 QoS 配置，一眼就能看出哪里不兼容。这条命令应该是你排查通信问题的第一反应。

实际开发中的常用搭配：

| 数据类型 | Reliability | Durability | History | 典型场景 |
|---------|------------|------------|---------|---------|
| 传感器数据 | Best Effort | Volatile | Keep Last(1) | /scan, /image_raw, /imu |
| 控制指令 | Reliable | Volatile | Keep Last(1) | /cmd_vel, /joint_commands |
| 地图数据 | Reliable | Transient Local | Keep Last(1) | /map, /costmap |
| TF 变换 | Best Effort | Volatile | Keep Last(1) | /tf |
| 参数/配置 | Reliable | Transient Local | Keep Last(1) | /parameter_events |

### CycloneDDS vs FastDDS - 两大实现怎么选

DDS 是一个标准，不是一个软件。就像 SQL 是标准，MySQL 和 PostgreSQL 是实现。ROS 2 支持多个 DDS 实现，目前最主流的两个是 **CycloneDDS** 和 **FastDDS**（原名 Fast-RTPS）。

**CycloneDDS** 由 Eclipse 基金会维护（前身来自 ADLINK）。从 ROS 2 Jazzy 版本开始，它是默认的 DDS 实现。特点是：

- 代码库小，启动快，资源占用低
- 延迟表现优秀，尤其是在数据量不大的场景下
- 配置相对简单，默认参数就能覆盖大部分场景
- 对共享内存（Shared Memory）传输有良好支持 - 同一台机器上的节点之间传数据可以绕过网络栈，直接在内存里交换，延迟极低

**FastDDS** 由 eProsima 开发。在 Humble 及更早版本中是默认实现。特点是：

- 功能更全面，QoS 策略支持更完整
- 大规模部署经验多 - 很多工业用户在生产环境里跑 FastDDS
- 配置灵活但也意味着配置项更多、更复杂
- 内存占用相对较高

**实际怎么选？** 如果你刚开始做机器人开发，或者你的场景是单机（一台机器人一台电脑），用 CycloneDDS - 它是新版本的默认选择，开箱即用，社区支持也好。如果你在做多机器人系统、跨网段通信、或者有特殊的 QoS 需求，FastDDS 的配置灵活性可能更适合。

切换 DDS 实现很简单，设一个环境变量就行：

```bash
# 使用 CycloneDDS
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp

# 使用 FastDDS
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
```

但有一个硬性约束：**同一个 ROS 2 网络里的所有节点必须使用同一种 DDS 实现。** CycloneDDS 的节点和 FastDDS 的节点互相看不见。如果你的机器人本体用了 FastDDS（因为厂商的 SDK 指定了），而你自己的算法节点默认是 CycloneDDS，它们之间就通不了信。这是另一个让人对着屏幕发呆半小时的经典问题 - "topic 名字对的，消息类型对的，QoS 也对的，为什么就是收不到？" 查一下两边的 `RMW_IMPLEMENTATION`。

### 实际开发中的通信问题

理论上 DDS 能搞定一切，但实际部署中通信层的问题五花八门。以下是最常见的几个。

**点云数据把带宽吃满了。** 一个 3D LiDAR 每秒产生的点云数据量可以到 100-200 MB。一个深度相机的点云也在几十 MB 量级。如果你的机器人上同时跑了 LiDAR、两个深度相机、还有高清 RGB 相机，总数据带宽可以轻松突破 500 MB/s。在千兆以太网（理论 125 MB/s）上这直接就爆了。

应对策略有几种：能不发就不发 - 点云做完处理（比如体素降采样、只提取关键特征）再发布，而不是把原始点云广播出去。用共享内存 - 如果发布者和订阅者在同一台机器上，配置 DDS 走共享内存传输可以完全绕过网络瓶颈。降低发布频率 - 很多时候你不需要 30Hz 的点云，10Hz 甚至 5Hz 就够了。

**多机器人通信的发现风暴。** DDS 的自动发现机制是通过组播（multicast）实现的 - 每个节点定期广播"我在这里"。当你有 5 台机器人，每台跑 30-50 个节点的时候，网络上会有 150-250 个节点同时在做自动发现。发现报文本身虽然小，但 DDS 还需要为每对可能的发布-订阅关系建立连接元数据，这个开销随节点数量平方级增长。

ROS 2 的解决方案是 **ROS_DOMAIN_ID** - 不同的 domain ID 让机器人之间的 DDS 通信隔离。同一个 domain 里的节点互相可见，不同 domain 的看不见。你给每台机器人分配一个不同的 domain ID（0-232 之间），它们就不会互相干扰。需要跨机器人通信的时候（比如调度系统和所有机器人对话），再单独搭一个桥接节点。

```bash
# 机器人 1
export ROS_DOMAIN_ID=1

# 机器人 2
export ROS_DOMAIN_ID=2
```

另一个更精细的方法是用 DDS 的 Discovery Server 模式（FastDDS 支持）- 不再用组播发现，而是指定一个服务器来集中管理节点发现。适合大规模车队部署。

**跨网段通信。** 机器人在仓库的 WiFi 上，你的开发机在办公室的有线网上，两个不同的子网 - DDS 的组播默认过不了路由器。这时候你需要配置 DDS 的"初始对端列表"（initial peers），手动告诉 DDS 对方的 IP 地址，让它用单播替代组播来发现节点。CycloneDDS 和 FastDDS 都支持通过 XML 配置文件来设置这些。

**消息时间戳不同步。** 多传感器融合（比如 LiDAR + IMU + 相机）需要对齐不同传感器的数据时间戳。如果你的各个传感器节点跑在不同的机器上，而这些机器的系统时钟没有同步，你融合出来的数据就是错的 - 用了 100ms 之前的 IMU 数据去配准当前帧的点云，SLAM 的精度会严重下降。解决方案是用 PTP（Precision Time Protocol）或者 NTP 做机器间的时钟同步，通常能把误差控制在微秒到毫秒级别。

### 什么时候你需要真正关心 DDS

说实话，大部分时间你不需要碰 DDS 的配置。ROS 2 的默认设置对单机器人、单台电脑的开发场景够用了。你该开始关心 DDS 的时刻通常是：

- 你的 `ros2 topic echo` 收不到数据，但 `ros2 topic list` 能看到 topic - 几乎一定是 QoS 不匹配或 DDS 实现不一致
- 你的系统开始出现明显的通信延迟 - 看看是不是大数据量话题（点云、图像）把带宽打满了
- 你开始部署多台机器人 - 是时候规划 domain ID 和网络拓扑了
- 你需要跨网段远程调试 - 需要配置 DDS 的发现策略

DDS 是那种"不出问题你感觉不到它存在，出了问题你必须理解它"的基础设施。就像你平时不关心 TCP/IP 协议栈，但网络出了问题的时候，`ping` 和 `traceroute` 是你的第一反应。对 ROS 2 开发者来说，`ros2 topic info --verbose`、检查 `RMW_IMPLEMENTATION` 和 `ROS_DOMAIN_ID` - 这三个是你排查 DDS 问题的基本工具。
